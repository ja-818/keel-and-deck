# Houston v2 Architecture -- Files-First Design

## Overview

Houston v2 replaces the SQLite-centric persistence model with a files-first approach.
Agent-visible data lives in `.houston/` workspace files. SQLite is only used for
conversation replay (chat_feed) and app preferences.

## The Rule

**If @houston-ai has a component that renders it -> `.houston/` folder**
**If it's app-specific -> `.houston/`, `.desktopclaw/`, etc.**

## Workspace Convention

Every Houston app project has this structure:

```
~/Documents/{AppName}/{ProjectName}/
  .houston/
    tasks.json
    routines.json
    channels.json
    goals.json
    skills/
      research.md
      writing.md
    log.jsonl
    config.json
```

## File Formats

### tasks.json
```json
[
  {
    "id": "uuid",
    "title": "Research AI safety frameworks",
    "description": "Survey top 5 frameworks, summarize differences",
    "status": "running",
    "claude_session_id": "960cb8e6-bcb6-..."
  }
]
```
Status values: `"queue"`, `"running"`, `"needs_you"`, `"done"`, `"cancelled"`

### routines.json
```json
[
  {
    "id": "uuid",
    "name": "Daily email digest",
    "description": "Summarize unread emails every morning",
    "trigger_type": "daily",
    "trigger_config": { "time": "09:00" },
    "status": "active",
    "approval_mode": "auto_approve",
    "claude_session_id": null
  }
]
```

### channels.json
```json
[
  {
    "id": "tg-main",
    "type": "telegram",
    "name": "Main Telegram",
    "token": "bot123:ABC..."
  }
]
```
Runtime state (connected/disconnected, message count) stays in memory, not in the file.

### goals.json
```json
[
  {
    "id": "uuid",
    "title": "Launch product by Q2",
    "status": "active"
  }
]
```

### skills/*.md
Markdown files with `## Instructions` and `## Learnings` sections.
One file per skill. Already exists in Houston, moving from `.houston/skills/` to `.houston/skills/`.

### log.jsonl
Append-only session audit trail. One JSON object per line.
```json
{"session_id":"960cb8e6","task_id":"abc-123","status":"completed","duration_ms":15000,"cost_usd":0.03,"timestamp":"2026-04-04T02:20:00Z"}
```

### config.json
Project-level settings (replaces DB preferences for project scope).
```json
{
  "name": "My Project",
  "claude_model": "sonnet",
  "claude_effort": "medium"
}
```

## SQLite (minimal -- app infrastructure only)

### chat_feed table (KEPT, modified)
```sql
CREATE TABLE chat_feed (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  claude_session_id TEXT NOT NULL,
  feed_type TEXT NOT NULL,
  data_json TEXT NOT NULL,
  source TEXT DEFAULT 'desktop',
  timestamp TEXT DEFAULT (datetime('now'))
);
CREATE INDEX idx_chat_feed_session ON chat_feed(claude_session_id);
```
Key change: keyed by `claude_session_id` (not project_id + feed_key).
This is the ONLY data table. Used for UI conversation replay on app restart.

### preferences table (KEPT, unchanged)
```sql
CREATE TABLE preferences (
  key TEXT PRIMARY KEY,
  value TEXT
);
```
App-level settings (last_project_id, etc). Not project-scoped.

### REMOVED tables
- issues -> tasks.json
- issue_dependencies -> removed
- issue_feed_items -> merged into chat_feed
- sessions -> log.jsonl
- session_events -> log.jsonl
- routines -> routines.json
- routine_runs -> log.jsonl + chat_feed
- routine_run_feed -> merged into chat_feed
- event_log -> log.jsonl
- webhooks -> removed
- channels -> channels.json
- memories -> removed entirely
- memories_fts -> removed entirely
- Houston-specific: goals -> goals.json, skills -> .houston/skills/, context_fields -> .houston/context.md, feedback_log -> removed, houston_conversations -> removed, conversation_messages -> merged into chat_feed

## Rust Module: houston_tauri::workspace_store

New module that provides typed read/write for `.houston/` files.

```rust
pub struct WorkspaceStore {
    root: PathBuf,  // e.g., ~/Documents/Houston/MyProject
}

impl WorkspaceStore {
    pub fn new(project_folder: &Path) -> Self;
    fn houston_dir(&self) -> PathBuf;  // root/.houston/

    // Tasks
    pub fn list_tasks(&self) -> Result<Vec<Task>>;
    pub fn create_task(&self, title: &str, description: &str) -> Result<Task>;
    pub fn update_task(&self, id: &str, updates: TaskUpdate) -> Result<Task>;
    pub fn delete_task(&self, id: &str) -> Result<()>;

    // Routines
    pub fn list_routines(&self) -> Result<Vec<Routine>>;
    pub fn create_routine(&self, routine: NewRoutine) -> Result<Routine>;
    pub fn update_routine(&self, id: &str, updates: RoutineUpdate) -> Result<Routine>;
    pub fn delete_routine(&self, id: &str) -> Result<()>;

    // Goals
    pub fn list_goals(&self) -> Result<Vec<Goal>>;
    pub fn create_goal(&self, title: &str) -> Result<Goal>;
    pub fn update_goal(&self, id: &str, updates: GoalUpdate) -> Result<Goal>;
    pub fn delete_goal(&self, id: &str) -> Result<()>;

    // Channels
    pub fn list_channels(&self) -> Result<Vec<ChannelEntry>>;
    pub fn add_channel(&self, channel: NewChannel) -> Result<ChannelEntry>;
    pub fn remove_channel(&self, id: &str) -> Result<()>;

    // Skills
    pub fn list_skills(&self) -> Result<Vec<Skill>>;
    pub fn read_skill(&self, name: &str) -> Result<Skill>;
    pub fn write_skill(&self, name: &str, instructions: &str, learnings: &str) -> Result<()>;
    pub fn delete_skill(&self, name: &str) -> Result<()>;

    // Log
    pub fn append_log(&self, entry: LogEntry) -> Result<()>;
    pub fn read_log(&self) -> Result<Vec<LogEntry>>;

    // Config
    pub fn read_config(&self) -> Result<ProjectConfig>;
    pub fn write_config(&self, config: &ProjectConfig) -> Result<()>;

    // Generic helpers
    fn read_json<T: DeserializeOwned>(&self, filename: &str) -> Result<T>;
    fn write_json<T: Serialize>(&self, filename: &str, data: &T) -> Result<()>;
    pub fn ensure_houston_dir(&self) -> Result<()>;  // creates .houston/ if missing
}
```

All operations use file locking (flock) to prevent concurrent write corruption.

## Tauri Commands: houston_tauri::workspace_commands (updated)

Existing file commands (list_project_files, open_file, etc.) stay.
New commands added for workspace_store operations:

```rust
// Tasks
#[tauri::command] pub async fn list_tasks(state, project_id) -> Vec<Task>;
#[tauri::command] pub async fn create_task(state, project_id, title, description) -> Task;
#[tauri::command] pub async fn update_task(state, project_id, task_id, updates) -> Task;
#[tauri::command] pub async fn delete_task(state, project_id, task_id) -> ();

// Routines, Goals, Channels, Skills -- same pattern

// Chat feed (still SQLite-backed)
#[tauri::command] pub async fn load_chat_feed(state, claude_session_id) -> Vec<Value>;
```

Note: `load_chat_feed` changes signature -- keyed by `claude_session_id` not `project_id`.

## What Gets Deleted

### houston (library)
- `crates/houston-cli/` -- entire crate removed from workspace
- `houston-db` tables: issues, issue_dependencies, issue_feed_items, sessions, session_events, routines, routine_runs, routine_run_feed, event_log, webhooks, channels, projects
- `houston-db` repos: repo_issues.rs, repo_issue_deps.rs, repo_projects.rs, repo_routines.rs, repo_routine_runs.rs, repo_session_events.rs
- `houston-memory/` -- entire crate removed from workspace
- `houston-events/` -- evaluate if still needed (event queue may be replaceable)
- `houston-tauri` modules that reference deleted tables

### houston (app)
- `houston_auto_spawn.rs` -- no more CLI parsing
- `houston_task_spawn.rs` -- tasks started directly, not via auto-spawn
- `houston_prompt.rs` -- simplified, no houston CLI instructions
- `houston_monitor.rs` -- simplified
- `houston_runner.rs` -- replaced with simpler session start
- `houston-shared` DB repos for deleted tables
- Frontend: IssueStore, ConversationStore, GoalStore (replaced by file-backed stores)
- Frontend: Houston planning chat panel (tasks ARE the conversations)

### desktopclaw
- Remove any houston-db issue/routine CRUD commands (use workspace_store)
- Simplify stores
