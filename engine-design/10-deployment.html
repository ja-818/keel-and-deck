Deployment: one binary, two homes.

Same Linux engine binary. Three deployment surfaces (desktop, self-hosted, Cloud). Almost zero divergence. The only difference is whose computer the Linux runtime runs on, and whether there's a control plane in front.

The engine binary is byte-for-byte the same in all three.

Desktop mode

The Linux engine binary, inside the host's Linux runtime (Chapter 3). Desktop shell talks to it over local networking on a loopback port. No control plane, no relay.

Self-host mode (Always On)

The Linux engine binary, native on a Linux VPS. Wrapped in Docker (always-on/Dockerfile) or systemd (always-on/houston-engine.service). The user puts a reverse proxy in front and binds HOUSTON_BIND_ALL=1. Already shipped as scaffolding for power users; see always-on/README.md.

Cloud mode (Houston Cloud)

The Linux engine binary, inside a Fly Machine per customer. A control plane in front handles signup, billing, admin, and provisioning. The relay routes inbound user traffic to the right machine and wakes sleeping machines on demand.

Why Fly Machines

Fly Machines are already Linux microVMs with first-class start and stop APIs, per-region placement, and sub-second wake. We can move to another platform later, but committing now means we can ship. Hedging delays decisions; pick one and move.

The control plane

Cloud is not just an engine on a VM. We need:

  • Signup and billing. Supabase Auth (already wired for desktop SSO) + Stripe for subscriptions.
  • Provisioning. A small service that, on first login, creates a Fly Machine for the customer, mints their bearer token, and writes initial workspace state.
  • Admin. Support tools: see a customer's recent errors, restart their machine, rotate their token.
  • Observability. Sentry (already used in desktop) gets a Cloud-scoped project. Per-machine log retention via Fly's built-in log shipping or BetterStack.

Lives at cloud/ (new). Probably TypeScript on Cloudflare Workers + Supabase for parity with the rest of our stack.

Storage when machines sleep across hosts

Fly volumes are pinned to a host. If a customer machine sleeps and wakes on a different physical host, the volume needs to follow. Two paths:

  • Pinned host (default). Each customer's machine has one home host. Wake is fast, no copy needed. Cost: that host becomes a single point of failure for that customer.
  • Cold storage in R2/S3 (failover). A nightly snapshot of the volume into object storage. If the home host is unavailable, we restore on a new host. Cost: ~30 seconds of cold-start when failover triggers.

Ship with pinned host. Add the snapshot path the moment we have paying customers.

Data residency

Each customer picks a region at signup (us-east, eu-west, ap-south). Their Fly Machine and snapshots live there. The control plane is global; only metadata (account, billing) lives outside the customer's region.

Backups

Nightly snapshot of the volume. Restore points: last 7 daily, last 4 weekly. Self-service restore from the customer dashboard.

Sleeping engines

For Cloud mode, engines sleep after N idle minutes (default 10). They exit 0 gracefully. The relay holds the connection from the next inbound request, calls Fly's start API, polls /v1/health, then proxies through. Cold start target: ~2 seconds. Acceptable for crons and webhooks; for active human chats, engines stay sticky-hot during a session window.

All four trigger sources from Chapter 9 (chat, mobile, Slack inbound, webhook) share the same wake-up path. Cron uses a slightly different path: a central scheduler service decides when to wake a machine, instead of the relay reacting to inbound traffic.

Pricing

Per VM-hour, or per agent-month. Either aligns with cost. The Cloud product is "rent a Fly Machine that runs Houston Engine plus our control plane around it." Self-hosters pay nothing to Houston and run the same engine themselves. Desktop users pay nothing and run the engine on their own laptop. All three meters touch the same engine binary.

The killer property

You can no longer have a "works on my Mac" bug, because there is no Mac engine. Every engine is the same Linux engine. The desktop just hosts the runtime.

What's there already

Always-on Docker recipe: always-on/Dockerfile and docker-compose.yml, builds the engine from source. Relay: houston-relay/, a production Cloudflare Worker + Durable Object at tunnel.gethouston.ai. Tunnel client crate: engine/houston-tunnel. What's missing: Linux runtime supervisor on desktop (Chapter 3), control-plane app at cloud/, idle-exit logic in the engine, wake API in the relay, Fly Machines integration.
