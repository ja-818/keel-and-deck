The roadmap.

Four milestones. Realistic sizes. Two of them run in parallel because they're independent. Two of them are heavier than they look. One parallel track starts on day one and has nothing to do with code (Apple paperwork).

The shape

M1 — Reliability (~4 weeks)

Goal: conversations never drop bytes. The "my message disappeared" failure mode goes away. Every byte the user sees is journaled before broadcast.

  • Add the four SQLite tables: turns, turn_stream, events_outbox, cron_runs. Concrete schemas in Chapter 6.
  • Write rows BEFORE doing work. Sweep orphans on boot.
  • Client-supplied turnId with idempotent server semantics. Protocol bump.
  • WebSocket replay by sequence number on reconnect. New since_seq on subscribe.
  • Cron audit log plus missed-fire catch-up policy.
  • Bounded event queue with per-source drop rules.
  • Coexistence with chat_feed: double-write for this release, drop in the next.

Mostly inserts and middleware. No architectural change. The 4-week number assumes the schema is locked on day one. Slips to 5-6 weeks if we discover the WS protocol bump rippling through more clients than expected.

M2 — Auth and credentials (~3 weeks, in parallel with M1)

Goal: Walls 2 and 3 from Chapter 7. No VM required. Owned by a different engineer so M1 and M2 ship as one release.

  • Typed tokens: (principal_id, scopes). New tables: principals and principal_tokens in houston-db.
  • require_scope(token, agent_path) middleware on every route that touches an agent.
  • Per-agent credentials: CLI spawn overrides HOME to a per-agent path under the agent's folder.
  • Migration: today's single device-bearer becomes one principal with full scope. No UX change for solo users.

Middleware plus a few new tables. About 3 weeks. Does not require Chapter 3, does not require a VM. Real isolation win shipped without waiting for M3.

M3 — Linux runtime everywhere (~3 months)

Goal: Wall 1 from Chapter 7. The whole desktop now runs the same Linux engine as Cloud. Per-agent Linux UIDs become possible.

  • Runtime supervisor at app/houston-tauri/runtime/.
  • macOS: Apple Virtualization.framework wrapper with a slim Linux guest image.
  • Windows: WSL2 detect/install/register/start.
  • Linux: passthrough, no runtime.
  • Per-agent useradd automation inside the runtime.
  • CLI spawn as hou_<agent>.
  • Import and Download routes for file flow in and out of the runtime.
  • Apple com.apple.security.virtualization entitlement in the signed app.

Honest scoping: this is two new pieces of infrastructure (a Linux guest image we build in CI, and a Tauri-side runtime supervisor) on top of changes throughout the engine to assume per-agent UIDs. Three months is the optimistic end. Could slip to four if VZ cold-start or WSL2 corp-IT issues bite.

M4 — Cloud foundation (~3 months)

Goal: Houston Cloud is shippable. Engine bounces don't kill conversations. Sleeping engines wake on demand. Paying customers can sign up.

  • Detached houston-turn-worker binary so engine restart doesn't kill in-flight turns. (The architectural risk piece.)
  • POST /v1/hooks/<route_token> wired to the event queue.
  • Engine learns to exit gracefully on idle.
  • Relay learns to wake a dormant engine via Fly Machines API.
  • Cron moves to a central scheduler service for hosted mode (in-engine cron stays for desktop and self-host).
  • Per-customer Fly Machine provisioning, region picker, onboarding.
  • Control plane: signup, billing, admin, observability. Lives at cloud/.
  • Storage strategy: pinned host with R2/S3 nightly snapshot fallback.

Heaviest milestone. Detached worker alone is 3-4 weeks if it goes well. Control plane is its own product. Three months is realistic only with two engineers on it; one engineer is closer to five.

Parallel track: Apple entitlement paperwork

Start day one of M1. Do not wait for M3. Using Virtualization.framework in a Developer ID app requires the com.apple.security.virtualization entitlement, which Apple grants by request. Timeline: a few weeks if clean, longer if Apple has questions. Filing late blocks the entire M3 release.

Action: open a ticket at developer.apple.com (Code Signing & Entitlements). Describe Houston as a desktop app that runs user-owned agent workloads inside a Linux guest for isolation. Reference VZ.framework. Wait. Then add the entitlement to app/src-tauri/entitlements.plist the moment it's granted.

Parallel track: VZ.framework cold-start spike

Before locking M3 dates, build a 1-week spike: bundle a minimal Alpine guest image with the engine binary, boot it via objc2-virtualization on a representative Mac, measure cold start (target ≤ 2s on Apple Silicon, ≤ 3s on Intel), measure resident memory (target ≤ 300MB), measure battery drain over a 4-hour idle session (target ≤ 10% extra vs native).

If the numbers come back outside those budgets, M3 needs a design rethink. Knowing this in week 0 is cheap. Discovering it in month 3 is not.

Open questions to settle before each milestone

Before M1: double-write or hard-migrate chat_feed in the first release? Recommendation: double-write.

Before M2: per-agent OAuth broker for desktop solo users (1 click, weaker iso) or strict per-agent (N clicks, stronger iso)? Recommendation: broker for desktop default, strict for Teams default.

Before M3: what's the fallback when a Windows machine has virtualization disabled in BIOS? Recommendation: clear install-time check, link to IT-admin docs, no native fallback (to keep one operating profile).

Before M4: pinned-host Fly Machines or active-active multi-region from day one? Recommendation: pinned + nightly snapshot, upgrade later. Also: where does the OSS-vs-Cloud line sit for the control plane? Cloud-only is the obvious answer.

What "shipped" looks like at the end of M4

  • Houston desktop on Mac, Windows, Linux, all running the same Linux engine inside a host runtime.
  • Per-agent isolation on every OS. The consultant-with-two-clients case is bulletproof.
  • Conversations survive engine bounces, app closes, network blips. Crons never silently drop.
  • Houston Cloud private beta: signup, per-customer Fly Machine, sleep/wake, billing, region picker.
  • Self-host story unchanged. docker pull && docker run still works.
  • Webhooks live. Five trigger sources (chat, mobile, Slack, cron, webhook) all routed through the same path.

Total elapsed: ~7-8 months from kickoff. Two engineers is enough for M1+M2 in parallel. Three is enough to run M3 and M4 in parallel after M2 lands. Anything less and the schedule stretches linearly.
