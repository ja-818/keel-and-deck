Chapter 05 Diagnosis · feeds M1

Why conversations die today.

The engine writes to disk only after the work is done. Everything in flight lives in memory. In-memory means it dies when the process dies, when the user closes the app, when the network blips. Five failure modes. Same root cause.

Failure mode #1

User closes the app mid-stream. This is the most common one and it has nothing to do with crashes. The user clicks the red button, the app exits, the engine subprocess exits with it, the in-flight assistant tokens vanish. The user reopens Houston and the message is gone. Today this happens dozens of times a day in the dev loop alone.

Failure mode #2

Engine crashes between "accept message" and "spawn CLI". Your message disappears. No record anywhere. You click send again, the engine has no memory of the first attempt. Rare but ugly when it happens.

Failure mode #3

Engine crashes mid-stream. Whatever the assistant had already streamed: gone. Database has nothing. The user thinks the agent is broken.

Failure mode #4

WebSocket reconnects after a network blip. Server has no record of what events it pushed during the disconnect. The frontend has to refetch state and pray nothing slipped through.

Failure mode #5

Cron fires while engine is restarting. The 7am alarm rings, the engine is rebooting, the fire is dropped silently. No log, no alert, no catch-up. Verified in engine/houston-scheduler/src/cron_job.rs: missed crons fire once on next boot if the schedule is past-due, but there's no record of how many fires were missed.

Failure mode #6 (the one most people miss)

CLI subprocess hangs without exiting. Provider gateway flakes, network stalls, model returns nothing. The CLI sits there for 20 minutes. Engine has no timeout, no started_at heartbeat, no idea anything is wrong. UI shows "thinking" forever.

The pattern

Notice the common thread. The engine writes to the database after doing work, not before. So anything happening between "before" and "after" lives only in memory, and that memory belongs to a process that can die or a socket that can disconnect.

The fix is boring but life-changing. Flip the order. Add timestamps. Add sequence numbers. Sweep orphans on boot. Chapter 6 covers how.

"Why hasn't this been fixed yet?"

Because in the happy path it works. Engines don't crash often. The team shipped features. The pain falls on users in the long tail: someone with a flaky home network, someone who closes the app to take a call, someone whose engine got OOM-killed in dev. The redesign moves us from "works most of the time" to "designed for the times it doesn't." That's the line between a beta and a product.
